## Решение 1


### 1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день

Для этого подходит **полное (full) резервное копирование**, выполняемое ежедневно (например, ночью).  
- Каждая резервная копия содержит **все данные** на момент создания.
- Восстановление происходит **из одной копии**, что упрощает процесс.
- Минус: занимает много места и времени при больших объёмах данных.

> **Рекомендация**: использовать `pg_dump` (для логического бэкапа) или `pg_basebackup` (для физического), запускаемый через `cron` ежедневно.


### 1.2. Необходимо восстанавливать данные за час до предполагаемой поломки

Здесь требуется **точечное восстановление (Point-in-Time Recovery, PITR)**.  
Для этого нужно:
1. **Полная резервная копия** (базовая точка восстановления).
2. **Архивирование WAL-файлов** (Write-Ahead Logging) — они содержат все изменения после полной копии.

С помощью этих двух компонентов можно восстановить состояние базы **на любую точку во времени**, включая «час назад».

> **Рекомендация**:  
> - Настроить `archive_mode = on` в PostgreSQL,  
> - Использовать `archive_command` для копирования WAL-файлов в безопасное место,  
> - При восстановлении указать `recovery_target_time` в файле `recovery.conf` (или `postgresql.auto.conf` в новых версиях).


### 1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных?

Да, такой сценарий возможен и реализуется через **высокую доступность (High Availability, HA)**.  
Основные подходы:

- **Репликация в реальном времени** (например, **streaming replication** в PostgreSQL):  
  Основной сервер (primary) непрерывно отправляет изменения на резервный (standby).

- **Механизм автоматического failover**:  
  Системы вроде **Patroni + etcd/Consul**, **repmgr** или **pg_auto_failover** отслеживают состояние primary-сервера.  
  При его отказе — автоматически **повышают standby до primary**, и приложения переключаются на него (часто через балансировщик, например, HAProxy).

- **Время переключения**: может составлять **от нескольких секунд до минуты**, в зависимости от настроек.

> Такой подход **не заменяет резервное копирование**, но **минимизирует простои**.  
> Резервные копии всё равно нужны на случай логических ошибок (например, `DROP DATABASE`), которые репликация тоже распространит.


### Вывод

- Для **ежедневного восстановления** — достаточно полных бэкапов.
- Для **точечного восстановления** — нужна комбинация full backup + WAL-архивирование (PITR).
- Для **мгновенного переключения** — требуется архитектура HA с репликацией и автоматическим failover.

## Решение 2

### 2.1. Пример команды резервного копирования и восстановления БД (`pg_dump` / `pg_restore`)

Согласно официальной документации PostgreSQL, для логического резервного копирования используются утилиты `pg_dump` (создание дампа) и `pg_restore` (восстановление из архива).

### Резервное копирование

Пример команды для создания сжатой резервной копии в формате `custom`:

```bash
pg_dump -U username -h localhost -F c -b -v -f backup_file.dump my_database

```
Где:

**-U** username — имя пользователя СУБД;

**-h** localhost — хост (можно опустить при локальном подключении);

**-F** c — указывает формат custom (бинарный, сжатый, поддерживает выборочное восстановление);

**-b** — включает large objects (если они используются);

**-v** — включает подробный вывод;

**-f** backup_file.dump — путь и имя файла резервной копии;

**my_database** — имя резервируемой базы данных
