## Решение 1


### 1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день

Для этого подходит **полное (full) резервное копирование**, выполняемое ежедневно (например, ночью).  
- Каждая резервная копия содержит **все данные** на момент создания.
- Восстановление происходит **из одной копии**, что упрощает процесс.
- Минус: занимает много места и времени при больших объёмах данных.

> **Рекомендация**: использовать `pg_dump` (для логического бэкапа) или `pg_basebackup` (для физического), запускаемый через `cron` ежедневно.


### 1.2. Необходимо восстанавливать данные за час до предполагаемой поломки

Здесь требуется **точечное восстановление (Point-in-Time Recovery, PITR)**.  
Для этого нужно:
1. **Полная резервная копия** (базовая точка восстановления).
2. **Архивирование WAL-файлов** (Write-Ahead Logging) — они содержат все изменения после полной копии.

С помощью этих двух компонентов можно восстановить состояние базы **на любую точку во времени**, включая «час назад».

> **Рекомендация**:  
> - Настроить `archive_mode = on` в PostgreSQL,  
> - Использовать `archive_command` для копирования WAL-файлов в безопасное место,  
> - При восстановлении указать `recovery_target_time` в файле `recovery.conf` (или `postgresql.auto.conf` в новых версиях).


### 1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных?

Да, такой сценарий возможен и реализуется через **высокую доступность (High Availability, HA)**.  
Основные подходы:

- **Репликация в реальном времени** (например, **streaming replication** в PostgreSQL):  
  Основной сервер (primary) непрерывно отправляет изменения на резервный (standby).

- **Механизм автоматического failover**:  
  Системы вроде **Patroni + etcd/Consul**, **repmgr** или **pg_auto_failover** отслеживают состояние primary-сервера.  
  При его отказе — автоматически **повышают standby до primary**, и приложения переключаются на него (часто через балансировщик, например, HAProxy).

- **Время переключения**: может составлять **от нескольких секунд до минуты**, в зависимости от настроек.

> Такой подход **не заменяет резервное копирование**, но **минимизирует простои**.  
> Резервные копии всё равно нужны на случай логических ошибок (например, `DROP DATABASE`), которые репликация тоже распространит.


### Вывод

- Для **ежедневного восстановления** — достаточно полных бэкапов.
- Для **точечного восстановления** — нужна комбинация full backup + WAL-архивирование (PITR).
- Для **мгновенного переключения** — требуется архитектура HA с репликацией и автоматическим failover.

## Решение 2

### 2.1. Пример команды резервного копирования и восстановления БД (`pg_dump` / `pg_restore`)

Согласно официальной документации PostgreSQL, для логического резервного копирования используются утилиты `pg_dump` (создание дампа) и `pg_restore` (восстановление из архива).

### Резервное копирование

Пример команды для создания сжатой резервной копии в формате `custom`:

```bash
pg_dump -U username -h localhost -F c -b -v -f backup_file.dump my_database

```
Где:

**-U** username — имя пользователя СУБД;

**-h** localhost — хост (можно опустить при локальном подключении);

**-F** c — указывает формат custom (бинарный, сжатый, поддерживает выборочное восстановление);

**-b** — включает large objects (если они используются);

**-v** — включает подробный вывод;

**-f** backup_file.dump — путь и имя файла резервной копии;

**my_database** — имя резервируемой базы данных.

### Восстановление

Для восстановления из файла в формате `custom` используется команда:

```bash
pg_restore -U username -h localhost -d new_database -v backup_file.dump

```

Где:

**-d** new_database — имя целевой базы данных (должна быть создана заранее!);
остальные параметры аналогичны.

### 2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

Да, процесс резервного копирования и управления архивами полностью автоматизируем.

1. Подход к автоматизации
- Создание Bash-скрипта, который:
- Выполняет pg_dump;
- Сохраняет бэкап с временной меткой;
- Удаляет старые копии (например, старше 7 дней);
- Логирует результат выполнения.
2. Настройка планировщика задач (cron) для запуска скрипта по расписанию.

**Пример скрипта `backup.sh`**:

```bash
#!/bin/bash

# Настройки
BACKUP_DIR="/backup/postgres"
DATE=$(date +%Y-%m-%d)
DB_NAME="my_database"
USER="postgres"

# Создание директории для бэкапов
mkdir -p "$BACKUP_DIR"

# Выполнение резервного копирования
pg_dump -U "$USER" -F c -b -v -f "$BACKUP_DIR/${DB_NAME}_$DATE.dump" "$DB_NAME"

# Удаление копий старше 7 дней
find "$BACKUP_DIR" -name "${DB_NAME}_*.dump" -mtime +7 -delete

# Запись в лог
echo "[$(date)] Backup completed: ${DB_NAME}_$DATE.dump" >> "$BACKUP_DIR/backup.log"

```

**Настройка cron**

Добавить задание в cron для ежедневного запуска в 02:00:

```
0 2 * * * /path/to/backup.sh
```

### 3.1. Пример команды инкрементного резервного копирования базы данных MySQL

В MySQL **инкрементное (дифференциальное) резервное копирование** реализуется через **бинарные логи (binary logs)**. Сама утилита `mysqldump` не поддерживает инкрементные бэкапы напрямую, но позволяет создать **полную резервную копию**, а последующие изменения фиксируются в binary log.

### Шаг 1. Включить binary logging (в `my.cnf` или `my.ini`)
```ini
[mysqld]
log-bin=mysql-bin
server-id=1
```
После перезапуска MySQL будут записываться все изменения в файлы `mysql-bin.000001`, `mysql-bin.000002` и т.д.

### Шаг 2. Сделать полную резервную копию (базовая точка)

```
mysqldump --single-transaction --flush-logs --master-data=2 -u root -p my_database > full_backup_$(date +%F).sql
```

- --single-transaction — обеспечивает согласованность для InnoDB без блокировки.
- --flush-logs — закрывает текущий binary log и создаёт новый.
- --master-data=2 — записывает позицию binary log в комментарии дампа (нужно для восстановления).

### Шаг 3. Инкрементное резервное копирование = копирование binary log файлов
После полной копии все изменения сохраняются в binary log. Чтобы сделать «инкрементный бэкап», достаточно скопировать эти файлы:

```
cp /var/log/mysql/mysql-bin.* /backup/mysql/
```

### Шаг 4. Восстановление

1. Восстановить полную копию:

```
mysql -u root -p my_database < full_backup_2025-01-25.sql
```

2. Применить изменения из binary log:

```
mysqlbinlog /backup/mysql/mysql-bin.000002 | mysql -u root -p
```

### 3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

Использование реплики (replica) даёт преимущества в следующих сценариях:

1. Минимизация времени простоя при отказе (High Availability)
При падении основного сервера можно мгновенно переключиться на реплику.
Резервное копирование не обеспечивает мгновенного переключения — восстановление может занять часы.
2. Разгрузка основного сервера
Реплика может обслуживать только чтение (например, отчёты, аналитика), снижая нагрузку на primary.
Резервное копирование (mysqldump) создаёт нагрузку на основной сервер во время выполнения.
3. Более частые точки восстановления
Репликация работает в реальном времени (с задержкой в секунды).
Резервные копии обычно делаются раз в день/час, что увеличивает возможную потерю данных.
4. Тестирование и разработка
Реплику можно использовать для тестирования обновлений, без риска повредить production-данные.

 ### Однако:
 
Реплика не заменяет резервное копирование!
Если на основном сервере произошла логическая ошибка (например, DROP DATABASE), она автоматически применится и на реплике.
Поэтому резервные копии всё равно обязательны — они позволяют откатиться до момента до ошибки.

 ### Вывод:
 
Инкрементное резервное копирование в MySQL строится на основе полного дампа + binary log.
Реплика даёт преимущество в скорости переключения, снижении нагрузки и актуальности данных, но не защищает от логических ошибок.
Идеальная стратегия: репликация + регулярные резервные копии.
