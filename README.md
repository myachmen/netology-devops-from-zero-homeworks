## Задание 1

Опишите основные преимущества использования масштабирования методами:

В качестве ответа в репозиторий шаблона с решением добавьте скриншоты с настройками раннера в проекте.

- активный master-сервер и пассивный репликационный slave-сервер;
- master-сервер и несколько slave-серверов;

*Дайте ответ в свободной форме.*


## Решение 1

**Активный master-сервер и пассивный slave-сервер**

В этой схеме все записи выполняются на master-сервере, а slave-сервер хранит его копию.

Преимущества:

- Повышается отказоустойчивость — при сбое master можно использовать slave.

- Можно делать резервные копии со slave, не нагружая master.

- Данные хранятся на нескольких серверах, что снижает риск их потери.

- Простая и понятная архитектура.

**Master-сервер и несколько slave-серверов**

Здесь master отвечает за запись данных, а slave-серверы используются для чтения.

Преимущества:

- Можно распределять запросы на чтение между slave-серверами.

- Снижается нагрузка на master-сервер.

- Повышается производительность системы.

- При отказе одного slave остальные продолжают работать.

- Архитектура легко масштабируется.





## Задание 2

Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:

- пользователи,
- книги,
- магазины (столбцы произвольно).

Опишите принципы построения системы и их разграничение или разбивку между базами данных.

*Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.*

## Решение 2

**1) Идея системы в целом**

Есть 3 таблицы: **users, books, shops**.

Задача — разнести данные так, чтобы:

- база **держала рост** (масштабирование),

- можно было **обслуживать чтение** большим числом реплик,

- запись шла **в правильное место** (в нужный шард).

Общий принцип:

- **вертикальный шардинг** - разделяем данные по смыслу (по “доменам”) на разные БД;

- **горизонтальный шардинг** - одну и ту же таблицу (обычно users) делим по ключу на несколько частей (шардов).

**2) Вертикальный шардинг (разделение по БД)**

Вариант разбиения по “сервисам/доменам”

**DB_Users**

- users (данные пользователей)

**DB_Catalog**

- books (каталог книг)

**DB_Shops**

- shops (каталог магазинов)

Зачем так делать:

- разные нагрузки: пользователи часто пишутся/меняются, каталог книг чаще читается, магазины тоже чаще читаются.

- можно масштабировать и обслуживать по-разному (например, users — сильнее защищаем и шардуем).

**3) Горизонтальный шардинг (на примере users)**

Таблица **users** обычно растёт быстрее всего, поэтому шардинг логично делать по ней.

**Принцип**

Делим users по user_id (или по hash(user_id)):

- **Shard U1**: user_id 1…10 000 000
- **Shard U2**: 10 000 001…20 000 000
- **Shard U3**: 20 000 001…30 000 000

(или consistent hashing / modulo: user_id % N)

Маршрутизация запросов:

- приложение или прокси (middleware) по user_id определяет, в какой шард идти.

*Важно: books и shops можно не шардировать, а держать как “справочники” (reference data) и масштабировать чтение репликами.*

**4) Режимы работы серверов (кто пишет/кто читает)**

**Для каждого шарда users**

- **Primary (master)** — принимает запись и важные транзакции.

- **Replica (slave)** — обслуживает чтение (read-only), разгружает primary.

- Реплик может быть несколько (для нагрузки и отказоустойчивости).

*Для catalog (books) и shops**

Обычно:

- **1 Primary** (редкие изменения: добавление книг/магазинов)

- несколько **Replica** для чтения (поиск, просмотр каталога)
