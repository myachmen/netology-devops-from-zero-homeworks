## Решение 1
 
### Задание 1. СУБД

Крупная строительная компания решает задачи по построению архитектуры работы с данными. Ниже — рекомендации по выбору типов СУБД для каждой предметной области.

---

#### **1.1. Бюджетирование проектов, финансовая аналитика и прогнозирование рисков**

**Рекомендуемая СУБД:** **Реляционная (SQL)** — например, **PostgreSQL** или **Microsoft SQL Server**.

**Обоснование:**  
Финансовые данные требуют строгой схемы, целостности и поддержки ACID-транзакций. Реляционные базы данных:
- обеспечивают надёжное хранение структурированных данных,
- поддерживают сложные JOIN-запросы и агрегации,
- позволяют строить аналитические отчёты и прогнозы.

PostgreSQL особенно подходит благодаря расширениям (например, `pg_partman`, `TimescaleDB`), а также встроенной поддержке материализованных представлений.

#### **1.1\*. Хеширование стало занимать длительное время — какое API можно использовать для ускорения?**

Если речь идёт о хешировании **чувствительных данных** (например, паролей), рекомендуется использовать криптографические библиотеки:
- **bcrypt**, **scrypt**, **Argon2** (через `passlib` в Python, `bcrypt` в Node.js и т.п.).

Если хеширование используется для **ускорения поиска** или **индексации**, возможны следующие решения:
- применение **Redis** как in-memory кэша,
- использование **хэш-индексов** в самой СУБД (например, `HASH` в PostgreSQL/MySQL),
- вынос тяжёлых вычислений в **OLAP-системы** (ClickHouse, Apache Doris) через ETL.

> Проблема, скорее всего, архитектурная: хеширование лучше выполнять на уровне приложения с помощью оптимизированных библиотек, а не в SQL.

---

### **1.2. Лендинги и CRM для сбора лидов**

**Рекомендуемая СУБД:**  
- Для **лендингов** — **документо-ориентированная NoSQL**, например **MongoDB**.  
- Для **CRM** — **PostgreSQL с типом `JSONB`** или также MongoDB (в зависимости от требований к целостности).

**Обоснование:**  
Данные лидов часто полуструктурированы и гетерогенны (разные формы, UTM-метки, источники). NoSQL-системы:
- гибко принимают новые поля без изменения схемы,
- обеспечивают высокую скорость записи и масштабируемость.

PostgreSQL с `JSONB` даёт «лучшее из двух миров»: гибкость документа + надёжность реляционной модели.

#### **1.2\*. Можно ли использовать одну СУБД?**

**Да.** Оптимальный выбор — **PostgreSQL**:
- лиды хранятся в таблице с колонкой типа `JSONB`,
- CRM-сущности (клиенты, сделки, этапы) — в реляционных таблицах,
- индексы GIN/GiST по `JSONB` обеспечивают быстрый поиск.

Альтернатива — MongoDB, но она менее подходит, если CRM требует строгих транзакций и отчётности.

---

### **1.3. База знаний: корпоративные нормы, обучающие материалы**

**Рекомендуемая СУБД:** **Реляционная (PostgreSQL)**.

**Обоснование:**  
Структура компании иерархична (департамент → отдел → сотрудник), а материалы организованы в категории. PostgreSQL поддерживает:
- рекурсивные CTE (`WITH RECURSIVE`) для обхода иерархий,
- чёткое управление доступом и версионированием.

#### **1.3\*. Можно ли использовать уже существующую СУБД?**

**Да.** Если выбран PostgreSQL для других задач, базу знаний можно реализовать в отдельной схеме (например, `knowledge_base`). Это:
- снижает эксплуатационную сложность,
- упрощает резервное копирование и мониторинг.

---

### **1.4. Логистика: маршруты доставки, распределение курьеров**

**Рекомендуемая СУБД:** **Графовая**, например **Neo4j** или **Amazon Neptune**.

**Обоснование:**  
Задачи маршрутизации — это задачи на **связи между сущностями** (объекты, склады, курьеры). Графовые базы:
- хранят связи напрямую,
- выполняют запросы «кратчайший путь», «все маршруты» за миллисекунды.

В реляционных СУБД такие запросы требуют множества JOIN’ов и становятся медленными при росте данных.

#### **1.4\*. Подключить отдел закупок?**

**Возможно, но с оговорками.**  
Если закупки моделируются как граф (поставщики → материалы → склады → объекты), их можно включить в ту же графовую БД.

Однако, если закупки требуют **финансовой точности** (см. 1.1), лучше:
- операционные данные — в графовой СУБД,
- финансовые итоги — в реляционной (PostgreSQL),
- синхронизировать через ETL-процессы.

---

### **1.5\*. Можно ли решить все задачи одной СУБД?**

**Теоретически — да. Практически — не рекомендуется для крупной компании.**

**Единственная универсальная СУБД — PostgreSQL**, благодаря:
- реляционной модели (финансы, база знаний),
- `JSONB` (лиды, гибкость),
- расширениям:  
  - **PostGIS** — для геомаршрутизации,  
  - **рекурсивные запросы** — для иерархий,  
  - **полноценный текстовый поиск** — для документов.

**Недостатки единой СУБД:**
- графовые задачи работают медленнее, чем в Neo4j,
- масштабируемость лендингов ограничена,
- сложность эксплуатации под разные нагрузки (OLTP, OLAP, документы).

**Рекомендация:**  
- Для **небольшой компании** — начать с **PostgreSQL как единой платформы**.  
- Для **крупной строительной компании** — использовать **поли-СУБД архитектуру**:
  - **PostgreSQL** — финансы, CRM, база знаний,
  - **MongoDB или PostgreSQL+JSONB** — лиды,
  - **Neo4j** — логистика,
  - **Redis** — кэширование.

Это обеспечит оптимальное сочетание **производительности**, **надёжности** и **гибкости** в каждой предметной области.

## Решение 2

### **2.1. Последовательность действий при успешном пополнении баланса счёта телефона (6 шагов)**

Транзакция — это логическая единица работы, которая должна быть выполнена **атомарно**, **согласованно**, **изолированно** и **надёжно** (ACID). При пополнении баланса телефона через платёжную систему должны произойти следующие действия:

1. **Инициация транзакции**  
   Система получает запрос от пользователя на пополнение (например, через мобильное приложение или веб-интерфейс). Начинается новая транзакция в базе данных.

2. **Проверка доступности средств и валидности данных**  
   Проверяются:
   - корректность номера телефона,
   - наличие достаточных средств на счёте пользователя (например, банковской карты),
   - лимиты и блокировки (антифрод, суточные лимиты и т.д.).

3. **Списание средств с источника оплаты**  
   С банковской карты (или другого источника) списывается указанная сумма. Это фиксируется как запись в таблице операций (например, `payment_transactions`).

4. **Начисление средств на баланс телефона**  
   В таблице пользовательских балансов (например, `user_balances`) обновляется значение баланса:  
   `UPDATE user_balances SET balance = balance + 500 WHERE phone_number = '+7...';`

5. **Фиксация транзакции (COMMIT)**  
   Если все предыдущие шаги выполнены успешно и без ошибок, транзакция подтверждается — изменения записываются в базу данных окончательно.

6. **Уведомление пользователя и логирование**  
   Пользователь получает подтверждение (SMS, push-уведомление), а система сохраняет аудит-лог операции для отчётности и мониторинга.

> Если на любом этапе (2–4) возникает ошибка — выполняется **откат транзакции (ROLLBACK)**, и баланс остаётся неизменным.

---

### **2.1\*. Пополнение через автоплатёж: особенности**

Если пополнение происходит **автоматически** (автоплатёж, например, по подписке или при балансе ниже порога), добавляются следующие действия:

1. **Запуск по расписанию или триггеру**  
   Система проверяет условия автоплатежа (например, через cron или фоновый worker):  
   - баланс ниже 100 руб.?  
   - наступил день списания по подписке?

2. **Автоматическая инициация транзакции без участия пользователя**  
   Платёж инициируется системой, а не человеком. Однако логика транзакции (списание → пополнение → commit) остаётся той же.

3. **Повышенные требования к обработке ошибок**  
   При неудаче система может:
   - повторить попытку через N часов,
   - уведомить пользователя о проблеме («не удалось продлить подписку»),
   - временно приостановить услугу.

4. **Хранение и проверка сохранённых платёжных данных**  
   Платёжные реквизиты (токен карты) хранятся в защищённом виде (PCI DSS-совместимо), и при автоплатеже используется **токен**, а не реальные данные карты.

> Остальные шаги (списание, пополнение, commit, логирование) аналогичны ручному пополнению — **транзакционная целостность сохраняется**.

---

Таким образом, как при ручном, так и при автоматическом пополнении, ключевым остаётся **сохранение ACID-свойств**, чтобы избежать рассогласования между платёжной системой и балансом пользователя.

## Решение 3

## 3.1. Пять преимуществ SQL-систем по сравнению с NoSQL

1. **Строгая схема данных и целостность (Data Integrity)**  
   SQL-системы требуют чёткого определения структуры таблиц заранее и поддерживают ограничения (constraints), внешние ключи и ссылочную целостность. Это минимизирует риск некорректных или «битых» данных.

2. **Поддержка ACID-транзакций**  
   Реляционные СУБД гарантируют выполнение транзакций по принципу ACID (атомарность, согласованность, изолированность, долговечность), что критически важно для финансовых, бухгалтерских и других систем, где недопустимы расхождения.

3. **Мощный и стандартизированный язык запросов (SQL)**  
   SQL — декларативный, хорошо документированный и универсальный язык, позволяющий выполнять сложные запросы: JOIN’ы, агрегации, оконные функции, рекурсивные запросы и т.д. Это упрощает разработку аналитики и отчётности.

4. **Зрелая экосистема и инструменты**  
   SQL-СУБД (PostgreSQL, MySQL, Oracle, SQL Server) имеют десятилетия развития, богатую документацию, встроенные средства резервного копирования, мониторинга, репликации, а также широкую поддержку в BI-системах (Tableau, Power BI и др.).

5. **Надёжность и предсказуемость**  
   Реляционные базы данных прошли проверку временем в enterprise-средах. Их поведение в условиях нагрузки, сбоев или конкурентного доступа хорошо изучено, что снижает риски при эксплуатации в критически важных системах.

---

## 3.1\*. Преимущества NewSQL-систем перед SQL и NoSQL

**NewSQL** — это новый класс СУБД, сочетающий преимущества реляционных (SQL) и масштабируемых (NoSQL) систем. Основные преимущества:

- **Горизонтальная масштабируемость + ACID-транзакции**  
  В отличие от классических SQL-систем (ограниченных вертикальным масштабированием), NewSQL (например, **CockroachDB**, **TiDB**, **Google Spanner**) может масштабироваться горизонтально, как NoSQL, **сохраняя при этом полную поддержку ACID-транзакций** даже между узлами.

- **Единая реляционная модель без компромиссов**  
  Разработчики работают с привычным SQL и схемами, как в PostgreSQL, но без необходимости вручную шардировать данные или жертвовать согласованностью ради производительности (что часто требуется в NoSQL).

- **Высокая доступность и отказоустойчивость «из коробки»**  
  NewSQL-системы обычно построены на распределённых архитектурах с автоматической репликацией, восстановлением и балансировкой, что снижает операционную нагрузку по сравнению с традиционными SQL-СУБД.

- **Совместимость с существующими приложениями**  
  Многие NewSQL-решения (например, TiDB) совместимы с протоколом MySQL, а CockroachDB — с PostgreSQL, что позволяет мигрировать существующие приложения с минимальными изменениями кода.

> **Итог:** NewSQL идеально подходит для современных облачных приложений, где нужны **масштабируемость**, **согласованность** и **реляционная модель** одновременно — без необходимости выбирать между «надёжностью» (SQL) и «гибкостью» (NoSQL).

## Выбор типа СУБД

Для задач, связанных с **массовыми вычислениями над большими объёмами данных** (Big Data), особенно если речь идёт об аналитике, машинном обучении, ETL-процессах или агрегациях, **традиционные реляционные СУБД (SQL)** не всегда оптимальны из-за:
- сложности горизонтального масштабирования,
- высоких накладных расходов на поддержание ACID в распределённой среде.

Вместо этого целесообразно использовать:
- **Распределённые NoSQL-системы** (например, **Apache Cassandra**, **ScyllaDB**, **MongoDB с шардированием**) — если данные полуструктурированы и важна высокая скорость записи/чтения.
- **OLAP-ориентированные СУБД** (например, **ClickHouse**, **Apache Doris**, **StarRocks**) — если задача — аналитика и агрегация.
- **Хранилища данных в формате «data lake»** (например, **Apache Hadoop HDFS + Parquet/ORC**) — если данные поступают в сыром виде и обрабатываются пакетно.

Однако, если задача требует **сложных JOIN’ов, транзакций и строгой целостности**, можно рассмотреть **распределённые NewSQL-системы** (например, **CockroachDB**, **TiDB**, **YugabyteDB**), но они лучше подходят для гибридных (HTAP) сценариев, а не для чисто вычислительных.

> **Вывод:** для вычислительно-ёмкой аналитики на 1000 узлах **лучше отказаться от классической СУБД как основного слоя хранения** и использовать **специализированные распределённые хранилища** (HDFS, S3, объектные хранилища) в связке с вычислительными фреймворками.

## Оптимальная модель распределённых вычислений

Наиболее подходящей моделью для описанного сценария является **модель MapReduce и её современные реализации**, в частности:

### **Apache Spark**

**Почему Spark?**
- Поддерживает **ин-мемори обработку**, что делает его **в 10–100 раз быстрее** классического Hadoop MapReduce.
- Работает с **огромными массивами данных**, распределяя вычисления по всем 1000 машинам.
- Поддерживает **разнообразные типы обработки**: пакетную, потоковую (streaming), машинное обучение (MLlib), графовые вычисления (GraphX).
- Совместим с различными источниками данных: HDFS, S3, Kafka, Parquet, JSON, CSV и даже JDBC-совместимыми СУБД.
- Обеспечивает **отказоустойчивость** за счёт пересчёта утраченных RDD/DataFrame-партиций.

Альтернативы:
- **Apache Flink** — если акцент на **реальном времени** (stream processing с низкой задержкой),
- **Dask** (в экосистеме Python) — для научных вычислений и ML на кластере,
- **Ray** — для распределённого выполнения задач, особенно в AI/ML.

## Итог

- **Критерий выбора**: тип рабочей нагрузки (OLTP vs OLAP vs batch analytics).
- **Тип СУБД**: для массовых вычислений — не классическая СУБД, а **распределённое файловое хранилище + колоночные/OLAP-форматы** (Parquet, ORC).
- **Модель вычислений**: **Apache Spark** — лучший выбор благодаря скорости, гибкости, отказоустойчивости и поддержке масштабирования на тысячи узлов.

Такая архитектура (например, **Spark + HDFS/MinIO + ClickHouse для результатов**) обеспечит максимальную эффективность при обработке больших данных на кластере из 1000 машин.
