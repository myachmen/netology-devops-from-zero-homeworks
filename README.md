## Задание 1. СУБД

Крупная строительная компания решает задачи по построению архитектуры работы с данными. Ниже — рекомендации по выбору типов СУБД для каждой предметной области.

---

### **1.1. Бюджетирование проектов, финансовая аналитика и прогнозирование рисков**

**Рекомендуемая СУБД:** **Реляционная (SQL)** — например, **PostgreSQL** или **Microsoft SQL Server**.

**Обоснование:**  
Финансовые данные требуют строгой схемы, целостности и поддержки ACID-транзакций. Реляционные базы данных:
- обеспечивают надёжное хранение структурированных данных,
- поддерживают сложные JOIN-запросы и агрегации,
- позволяют строить аналитические отчёты и прогнозы.

PostgreSQL особенно подходит благодаря расширениям (например, `pg_partman`, `TimescaleDB`), а также встроенной поддержке материализованных представлений.

#### **1.1\*. Хеширование стало занимать длительное время — какое API можно использовать для ускорения?**

Если речь идёт о хешировании **чувствительных данных** (например, паролей), рекомендуется использовать криптографические библиотеки:
- **bcrypt**, **scrypt**, **Argon2** (через `passlib` в Python, `bcrypt` в Node.js и т.п.).

Если хеширование используется для **ускорения поиска** или **индексации**, возможны следующие решения:
- применение **Redis** как in-memory кэша,
- использование **хэш-индексов** в самой СУБД (например, `HASH` в PostgreSQL/MySQL),
- вынос тяжёлых вычислений в **OLAP-системы** (ClickHouse, Apache Doris) через ETL.

> Проблема, скорее всего, архитектурная: хеширование лучше выполнять на уровне приложения с помощью оптимизированных библиотек, а не в SQL.

---

### **1.2. Лендинги и CRM для сбора лидов**

**Рекомендуемая СУБД:**  
- Для **лендингов** — **документо-ориентированная NoSQL**, например **MongoDB**.  
- Для **CRM** — **PostgreSQL с типом `JSONB`** или также MongoDB (в зависимости от требований к целостности).

**Обоснование:**  
Данные лидов часто полуструктурированы и гетерогенны (разные формы, UTM-метки, источники). NoSQL-системы:
- гибко принимают новые поля без изменения схемы,
- обеспечивают высокую скорость записи и масштабируемость.

PostgreSQL с `JSONB` даёт «лучшее из двух миров»: гибкость документа + надёжность реляционной модели.

#### **1.2\*. Можно ли использовать одну СУБД?**

**Да.** Оптимальный выбор — **PostgreSQL**:
- лиды хранятся в таблице с колонкой типа `JSONB`,
- CRM-сущности (клиенты, сделки, этапы) — в реляционных таблицах,
- индексы GIN/GiST по `JSONB` обеспечивают быстрый поиск.

Альтернатива — MongoDB, но она менее подходит, если CRM требует строгих транзакций и отчётности.

---

### **1.3. База знаний: корпоративные нормы, обучающие материалы**

**Рекомендуемая СУБД:** **Реляционная (PostgreSQL)**.

**Обоснование:**  
Структура компании иерархична (департамент → отдел → сотрудник), а материалы организованы в категории. PostgreSQL поддерживает:
- рекурсивные CTE (`WITH RECURSIVE`) для обхода иерархий,
- чёткое управление доступом и версионированием.

#### **1.3\*. Можно ли использовать уже существующую СУБД?**

**Да.** Если выбран PostgreSQL для других задач, базу знаний можно реализовать в отдельной схеме (например, `knowledge_base`). Это:
- снижает эксплуатационную сложность,
- упрощает резервное копирование и мониторинг.

---

### **1.4. Логистика: маршруты доставки, распределение курьеров**

**Рекомендуемая СУБД:** **Графовая**, например **Neo4j** или **Amazon Neptune**.

**Обоснование:**  
Задачи маршрутизации — это задачи на **связи между сущностями** (объекты, склады, курьеры). Графовые базы:
- хранят связи напрямую,
- выполняют запросы «кратчайший путь», «все маршруты» за миллисекунды.

В реляционных СУБД такие запросы требуют множества JOIN’ов и становятся медленными при росте данных.

#### **1.4\*. Подключить отдел закупок?**

**Возможно, но с оговорками.**  
Если закупки моделируются как граф (поставщики → материалы → склады → объекты), их можно включить в ту же графовую БД.

Однако, если закупки требуют **финансовой точности** (см. 1.1), лучше:
- операционные данные — в графовой СУБД,
- финансовые итоги — в реляционной (PostgreSQL),
- синхронизировать через ETL-процессы.

---

### **1.5\*. Можно ли решить все задачи одной СУБД?**

**Теоретически — да. Практически — не рекомендуется для крупной компании.**

**Единственная универсальная СУБД — PostgreSQL**, благодаря:
- реляционной модели (финансы, база знаний),
- `JSONB` (лиды, гибкость),
- расширениям:  
  - **PostGIS** — для геомаршрутизации,  
  - **рекурсивные запросы** — для иерархий,  
  - **полноценный текстовый поиск** — для документов.

**Недостатки единой СУБД:**
- графовые задачи работают медленнее, чем в Neo4j,
- масштабируемость лендингов ограничена,
- сложность эксплуатации под разные нагрузки (OLTP, OLAP, документы).

**Рекомендация:**  
- Для **небольшой компании** — начать с **PostgreSQL как единой платформы**.  
- Для **крупной строительной компании** — использовать **поли-СУБД архитектуру**:
  - **PostgreSQL** — финансы, CRM, база знаний,
  - **MongoDB или PostgreSQL+JSONB** — лиды,
  - **Neo4j** — логистика,
  - **Redis** — кэширование.

Это обеспечит оптимальное сочетание **производительности**, **надёжности** и **гибкости** в каждой предметной области.
